# 编排引擎运作模型

本文档用于统一 Rhythm 中“主 Agent + 编排引擎 + 多 Agent 执行”的概念、命名和整体运作过程。

目标不是描述一个静态模板工作流系统，而是定义一个可以在用户授权后，代替用户持续推进项目的长时运行系统。

## 1. 命名约定

### 主 Agent

特指会话区域中的 Agent。

它直接与用户对话，负责：

- 理解目标
- 澄清需求
- 生成计划文档
- 请求用户确认
- 在合适时机将任务移交给编排引擎

主 Agent 不负责长时间持续推进整个项目。

### 编排 Agent

特指 run 内部的调度 Agent。

它不直接干活，而是：

- 读取计划文档
- 读取任务文档
- 读取审核规则与审核记录
- 读取项目状态和结构性产物
- 决定下一步做什么
- 唤醒执行 Agent
- 唤醒审核 Agent
- 根据结果继续、返工、暂停或请求人工介入

编排 Agent 的职责本质上是“代替用户持续推进项目”。

### Work Agent

特指执行具体任务的 Agent。

它只处理局部工作，例如：

- 写某一部分小说大纲
- 拆解某一卷结构
- 设计角色关系
- 编写某个模块代码
- 补齐某个目录结构

Work Agent 只对当前任务负责，不负责全局推进。

### 审核 Agent

特指负责审核产出的 Agent。

它根据审核规则判断：

- 当前结果是否达标
- 是否需要返工
- 返工点是什么
- 是否可以进入下一阶段

审核 Agent 是质量门禁，不是可选装饰。

### 引擎

特指编排引擎本身。

引擎不负责业务思考，它的职责是：

- 持久化 run
- 持久化各类文档对象
- 唤醒编排 Agent
- 承载 Work Agent / 审核 Agent 运行
- 管理暂停、恢复、取消、继续推进
- 管理事件流、诊断、恢复和可视化

## 2. 系统目标

编排引擎存在的目的，是在用户确认方向之后，替代用户持续推动任务往前走。

传统模式通常是：

1. 用户与主 Agent 对话
2. 主 Agent 产出规划
3. 用户要求执行第一步
4. 主 Agent 做第一步
5. 用户继续要求执行下一步
6. 主 Agent 再继续推进

编排引擎模式则变成：

1. 用户与主 Agent 对话
2. 主 Agent 产出计划
3. 用户确认计划
4. 创建 run 并移交给编排引擎
5. 编排 Agent 根据计划和当前任务状态，持续唤醒 Work Agent / 审核 Agent
6. 引擎长期运行，直到项目完成、暂停、失败或人工介入

因此，编排引擎并不是简单“跑模板”，而是接管用户原本需要持续做的推进动作。

## 3. 核心文档对象

一份文档不足以支撑长时项目。系统需要至少维护以下几类文档对象。

### 3.1 `plan`

`plan` 由主 Agent 生成，并且必须经过用户确认。

它描述的是项目的总体意图和推进原则，而不是细粒度执行清单。

`plan` 至少应包含：

- 项目目标
- 背景与上下文
- 约束条件
- 期望输出
- 总体阶段划分
- 拆解原则
- 哪些节点需要审核
- 审核的大方向标准
- 哪些节点需要人工确认

`plan` 是整个 run 的高层依据。

关键原则：

- `plan` 不能由系统自己生成自己执行
- `plan` 必须先由主 Agent 生成
- `plan` 必须由用户确认后，才允许移交给编排引擎

### 3.2 `task`

`task` 是编排 Agent 自己维护的执行文档。

它不是一次性定死的待办列表，而是随着项目推进持续变化的任务树或任务图。

`task` 至少应表达：

- 当前有哪些待办项
- 当前优先级
- 已完成项
- 被阻塞项
- 可继续拆解的项
- 父子任务关系
- 返工项
- 待审核项
- 当前建议唤醒哪个 Agent

`task` 的特点：

- 动态变化
- 可反复拆解
- 可在运行中重排
- 必须始终受 `plan` 约束

例如“写小说大纲”这个大任务，可以逐步拆为：

- 大纲规划
- 卷级拆解
- 阶段拆解
- 章节拆解

这个过程不要求一开始就完全规划周全，而是由编排 Agent 按 `plan` 的拆解原则逐步展开。

### 3.3 审核文档

审核相关内容建议拆成两类：

#### `review policy`

审核规则文档，用于定义：

- 什么节点需要审核
- 每一步后审核还是阶段后审核
- 审核标准是什么
- 通过条件是什么
- 不通过时如何处理
- 是否允许人工 override

#### `review log`

审核记录文档，用于记录：

- 审核对象
- 审核时间
- 审核结论
- 不通过原因
- 返工意见
- 是否人工覆盖

审核文档的作用，是让“质量控制”成为 run 的一等机制。

### 3.4 结构性产物 / 项目状态

系统还需要持续维护结构性产物，也可以称为 `project state` 或 `artifacts`。

它描述的是项目已经形成的稳定事实，而不只是对话消息。

例如：

- 小说项目已经有哪些卷、阶段、章节
- 当前世界观设定有哪些稳定条目
- 当前角色设定有哪些稳定条目
- 软件项目当前的目录结构
- 已有模块和文件
- 已沉淀的设计文档、草稿、报告

这些结构性产物必须可以被编排 Agent 读取，并作为后续决策依据。

## 4. 四类角色之间的关系

### 主 Agent 与用户

主 Agent 负责理解用户，并生成高质量 `plan`。

### 用户与编排引擎

用户不直接维护 run 内的细节推进，而是在确认 `plan` 后授权编排引擎接管推进。

### 编排 Agent 与 Work Agent

编排 Agent 不自己干活，只根据 `plan + task + review + project state` 生成更高质量的任务说明，并动态发起 Work Agent。

### 编排 Agent 与审核 Agent

当某一步或某一阶段达到 `review policy` 定义的审核点时，编排 Agent 必须发起审核 Agent，而不是直接跳到下一步。

## 5. 完整运作过程

### 阶段 0：需求澄清

用户在主会话中与主 Agent 对话。

目标：

- 把需求说清楚
- 把范围说清楚
- 把约束说清楚
- 把希望的交付物说清楚

此时仍然是普通会话阶段。

### 阶段 1：主 Agent 生成 `plan`

在需求足够清楚后，主 Agent 生成一份计划文档。

这份文档不是静态模板实例，而是针对本次项目的高层计划。

它回答：

- 项目要做什么
- 计划怎么推进
- 可能分几层拆解
- 哪些节点需要审核
- 成功标准是什么

### 阶段 2：用户确认 `plan`

这是整个系统的关键闸门。

只有当用户确认 `plan` 后，才允许把任务移交给编排引擎。

原因：

- 避免系统自己生成、自己批准、自己执行
- 避免项目从一开始就在错误方向上长时间运行
- 让用户对目标、边界、拆解原则和审核原则拥有最终决定权

### 阶段 3：创建 run

用户确认 `plan` 后，系统创建 run。

run 是一个长期运行对象，至少绑定以下内容：

- 已确认的 `plan`
- 初始 `task`
- `review policy`
- `review log`
- `project state`
- 编排 Agent 状态
- Agent 运行记录
- 事件流

run 创建后，应在主会话中以可点击对象的形式出现。

### 阶段 4：引擎唤醒编排 Agent

引擎本身不直接做业务决策，它只是唤醒编排 Agent。

每次编排 Agent 被唤醒时，应按照类似顺序工作：

1. 读取 `plan`
2. 读取 `task`
3. 读取 `review policy`
4. 读取 `review log`
5. 读取 `project state`
6. 判断当前最合适的下一步

### 阶段 5：编排 Agent 动态发起 Work Agent

如果当前需要具体执行，则编排 Agent 生成高质量任务说明，并发起一个或多个 Work Agent。

这里有一个关键点：

用户给主 Agent 的消息通常是短语级的，而编排 Agent 可以基于 `plan` 和当前状态生成更完整、更明确的系统提示与模拟用户消息。

因此，编排 Agent 给 Work Agent 的任务说明通常会比真实用户直接下达的指令更好。

### 阶段 6：Work Agent 产出结果

Work Agent 完成局部任务后，会产出：

- 会话记录
- 局部结果
- 结构性产物更新
- 可能的任务拆解建议

这些内容回写给 run。

### 阶段 7：进入审核

Work Agent 产出后，是否进入审核、何时进入审核、按什么标准审核，都由 `review policy` 定义。

编排 Agent 不能跳过该门禁。

如果到达审核点，则发起审核 Agent。

### 阶段 8：审核 Agent 判断结果

审核 Agent 根据规则做出判断：

- 通过：允许进入下一步
- 不通过：生成返工意见，回到相应任务
- 无法判断：暂停并请求人工介入

审核记录必须写入 `review log`。

### 阶段 9：编排 Agent 更新 `task`

审核结束后，编排 Agent 更新动态任务文档：

- 标记已完成项
- 标记返工项
- 新建子任务
- 调整优先级
- 继续拆解
- 决定下一步执行对象

这个过程会持续重复，直到 run 完成。

### 阶段 10：人工介入

用户在任意阶段都应可以：

- 暂停 run
- 恢复 run
- 修改任务说明
- 跳过节点
- 重试节点
- 接管某个 Agent 会话
- 回到编排引擎继续运行

人工介入不会推翻整体模型，而是作为运行中的必要控制能力。

## 6. 编排 Agent 的工作原则

编排 Agent 需要遵循以下原则：

### 6.1 不自己干活

编排 Agent 不直接产出业务内容。

它的职责是推进，不是执行。

### 6.2 始终受 `plan` 约束

`task` 可以动态变化，但不能脱离 `plan` 的总体目标和拆解原则。

### 6.3 优先读文档状态，而不是读长对话历史

编排 Agent 的核心输入应是：

- `plan`
- `task`
- `review`
- `project state`

而不是依赖一条无限增长的聊天上下文。

### 6.4 审核必须成为硬门禁

如果 `review policy` 规定需要审核，则必须经过审核 Agent 或人工批准，不允许绕过。

### 6.5 可拆解，但不强求一开始拆到底

长项目不可能从第一刻起就规划得极其完备。

因此系统应支持：

- 先粗规划
- 运行中继续拆解
- 拆解结果回写 `task`

## 7. 模板在系统中的位置

模板仍然存在，但它的定位应弱于最终执行文档。

更准确地说：

- 模板是通用的 `plan scaffold`
- 当前 run 真正执行的是经用户确认后的 `plan`

因此模板不应再被理解为“固定流程本体”，而应被理解为：

- 一个常用项目类型的起始骨架
- 帮助主 Agent 更快生成 `plan`
- 帮助用户更快审阅和确认

模板可以存在，但系统的核心不应该是模板，而应该是：

- 已确认的 `plan`
- 动态维护的 `task`
- 明确的审核机制
- 可持续更新的 `project state`

## 8. 典型示例：小说大纲项目

用户目标：写一本长篇小说的大纲。

### 主会话阶段

1. 用户与主 Agent 聊清楚题材、风格、目标读者、篇幅等
2. 主 Agent 生成 `plan`
3. `plan` 中说明：
   - 先做概念定位
   - 再做世界观与角色
   - 再形成总纲
   - 再拆卷
   - 再拆阶段和章节
   - 每一大阶段结束后需要审核
4. 用户确认 `plan`

### run 阶段

1. 引擎创建 run
2. 编排 Agent 先创建初始 `task`
3. 编排 Agent 发起 Work Agent 做“概念定位”
4. 完成后进入审核
5. 审核通过后，编排 Agent 更新 `task`
6. 再发起 Work Agent 做“世界观设定”
7. 再做角色设定
8. 再审核
9. 再推进总纲、卷纲、章节结构

整个过程中：

- `plan` 提供方向
- `task` 提供当前执行图
- 审核文档保证质量
- 结构性产物记录已经沉淀下来的设定和大纲结构

## 9. 实现上的设计含义

基于以上模型，系统后续应逐步从“模板中心”转向“run 文档中心”。

更准确的重点应是：

- 主 Agent 生成并提交 `plan`
- 用户确认 `plan`
- run 绑定 `plan / task / review / project state`
- 编排 Agent 每次被唤醒时优先读取这些对象
- Work Agent 和审核 Agent 都只是 run 的执行单元

因此，编排引擎最终不是一个普通工作流编辑器，而是一个：

`基于已确认计划、通过动态任务和审核机制持续推进项目的代理执行系统`

## 10. 总结

整个系统的核心逻辑可以总结为：

1. 主 Agent 与用户对话并生成 `plan`
2. 用户确认 `plan`
3. 系统创建 run
4. 引擎唤醒编排 Agent
5. 编排 Agent 读取 `plan + task + review + project state`
6. 编排 Agent 发起 Work Agent 干活
7. 到达审核点后发起审核 Agent
8. 审核通过则继续，不通过则返工或暂停
9. 编排 Agent 持续更新 `task` 和项目状态
10. 引擎负责长时运行、恢复、诊断和可视化

换句话说：

主 Agent 负责“把事情定义清楚”。

编排 Agent 负责“代替用户持续推进”。

Work Agent 负责“把具体事情做出来”。

审核 Agent 负责“确保结果达标再继续”。
