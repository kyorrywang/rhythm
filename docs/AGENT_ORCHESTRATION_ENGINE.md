# Agent 编排引擎

## 1. 文档目的

本文档用于定义 Rhythm `Agent 编排引擎` 的产品边界、核心模型、工作方式、开发范围与验收标准。

这份文档不是概念介绍，而是后续开发与验收的统一依据。

当产品、交互、数据结构、运行时实现发生分歧时，以本文档为准。

## 2. 名称与定位

本系统名称固定为：

`Agent 编排引擎`

它的定位是：

- 一个通用的 Agent 编排与运行内核
- 一个可装载不同流程模板的长期任务执行系统
- 一个支持自然语言驱动与工作台驱动双入口的引擎

它不是：

- `n8n` 风格的自动化平台
- 以第三方工具节点生态为核心的编排器
- 软件开发专用系统

软件开发、小说创作、报告撰写等，都只是该引擎上的不同模板。

## 3. 要解决的问题

当前系统已经具备较完整的 Agent 基础设施，但仍缺少一层关键能力：

用户提出一个高层目标后，系统还不能稳定地：

- 选择适合的方法论模板
- 将目标转成长期运行的流程实例
- 按阶段持续推进
- 在关键节点进行精细化管控
- 支持暂停、恢复、查询、回退与重试
- 以工件而非纯对话的方式累计成果

例如，用户可能提出：

- 帮我复刻一套 Airtable
- 帮我开发一套 ERP
- 帮我把这个脑洞写成小说
- 帮我写一份某行业研究报告

系统需要做的不是单轮输出答案，而是：

1. 理解目标
2. 选择或生成模板
3. 启动流程实例
4. 按模板推进任务
5. 持续返回进展、产物、风险和待确认点

## 4. 核心定义

### 4.1 引擎

引擎负责：

- 模板注册与匹配
- 流程实例创建
- 运行状态管理
- 调度 Agent / sub-agent / tools
- 暂停、继续、取消、重试、恢复
- 工件记录
- 审计与事件日志

### 4.2 模板

模板是“做一类任务的方法论”。

例如：

- 软件开发模板
- 起点小说模板
- 番茄小说模板
- 调研报告模板

模板定义的是：

- 阶段如何拆分
- 每个阶段的目标
- 每个阶段的输入输出
- 每个阶段允许使用的能力
- 阶段通过条件
- 阶段失败后的处理方式

### 4.3 流程实例

流程实例是模板在一次具体目标上的运行结果。

例如：

- “复刻 Airtable” 对应一个流程实例
- “把某脑洞写成长篇小说” 对应另一个流程实例

流程实例必须能被持续追踪、暂停、恢复、查询和管理。

### 4.4 工件

工件是流程运行过程中沉淀的结构化成果。

例如：

- 需求说明
- 架构设计
- 章节大纲
- 角色设定
- 草稿
- 审核记录
- 测试报告

工件是引擎的重要输出，不应只依赖对话历史。

## 5. 工作方式

Agent 编排引擎必须支持两条同等重要的工作路径。

### 5.1 会话驱动

会话是自然语言入口。

用户可以在会话中：

- 讨论目标
- 澄清需求
- 参考已有模板生成新模板
- 启动流程
- 查询运行状态
- 管理运行中的流程

典型示例：

- 我有一个脑洞，想写成小说
- 帮我把这个脑洞写成小说
- 参考起点模板，帮我写一套番茄小说模板
- 列出现在运行中的流程
- 把刚才那个流程暂停
- 继续上次那个流程

在这条路径中，Agent 通过 tool 或 skill 调用编排引擎能力。

### 5.2 工作台驱动

`workflow` 页面不是唯一入口，但它是明确的管理入口。

它的定位是：

`模板与运行管理工作台`

工作台需要支持：

- 创建模板
- 编辑模板
- 查看模板列表
- 手动运行模板
- 查看运行中的流程
- 暂停、继续、取消流程
- 查看流程历史与状态

### 5.3 双入口一致性

会话入口和工作台入口必须操作同一套后端对象，而不是两套平行系统。

也就是说，这两条路径最终都应落到相同的数据与命令模型上：

- Template
- Process Instance
- Run State
- Artifacts
- Runtime Commands

## 6. 产品边界

### 6.1 一期必须支持

一期必须实现以下能力：

- 模板创建
- 模板编辑
- 模板保存
- 手动运行模板
- 从会话中启动模板
- 从会话中查询运行状态
- 从会话中执行基础管理动作
- 运行中的流程列表
- 流程暂停
- 流程继续
- 流程取消
- 基础运行历史查看
- 模板驱动的长期运行
- LLM 为核心节点的模板执行能力

### 6.2 一期建议支持

如果成本可控，一期建议补齐：

- 从会话中生成新模板草稿
- 模板克隆
- 模板版本字段
- 失败节点重试
- 流程实例的工件列表展示

### 6.3 一期明确不做

以下内容不作为一期必需项：

- 大规模第三方 SaaS 集成
- 复杂并行调度
- 完整低代码生态
- 完整表达式语言
- 通用节点市场
- 高级权限系统

## 7. 核心对象模型

下面的模型是开发时的统一抽象。

### 7.1 Template

模板描述一类流程。

建议字段：

```ts
interface AgentTemplate {
  id: string;
  name: string;
  domain: string;
  description?: string;
  version: string;
  stages: AgentStage[];
  metadata?: Record<string, unknown>;
}
```

### 7.2 Stage

阶段描述模板中的一个步骤域。

```ts
interface AgentStage {
  id: string;
  name: string;
  goal: string;
  workflowId?: string;
  inputs?: string[];
  outputs?: string[];
  gates?: AgentGate[];
}
```

### 7.3 Process Instance

流程实例描述某次任务运行。

```ts
interface AgentProcessInstance {
  id: string;
  templateId: string;
  goal: string;
  status: 'pending' | 'running' | 'paused' | 'completed' | 'failed' | 'cancelled';
  currentStageId?: string;
  currentRunId?: string;
  variables: Record<string, unknown>;
  artifacts: Record<string, unknown>;
  history: AgentProcessEvent[];
}
```

### 7.4 Gate

门禁描述阶段是否允许继续推进。

```ts
interface AgentGate {
  id: string;
  name: string;
  type: 'artifact' | 'approval' | 'assertion' | 'review';
  rule: string;
}
```

## 8. 当前 workflow 的定位

当前 `workflow` 能力不应再被理解为最终产品本体，而应被理解为：

- 模板编辑与管理的工作台
- 模板底层 DSL 的承载层
- 阶段内部执行逻辑的表达方式

换句话说：

- `Agent 编排引擎` 是上层产品对象
- `workflow` 是它的一部分实现与管理界面

因此，后续命名和交互上应弱化“workflow 是主产品”的心智，强化“workflow 是模板工作台”的心智。

## 9. 会话中的模板生成

这是本系统的关键差异化能力之一。

模板不应只支持手工创建，还应支持通过会话协作生成。

典型链路：

1. 用户已有一个“起点小说模板”
2. 用户在会话中要求参考该模板生成“番茄小说模板”
3. Agent 对比已有模板与目标风格差异
4. Agent 输出新的模板草稿
5. 模板进入工作台供继续编辑、调整和发布

因此，系统需要支持：

- 根据已有模板生成新模板
- 从会话中创建模板
- 将会话生成结果落到模板存储中
- 工作台继续编辑该模板

## 10. 运行状态管理

运行状态管理必须同时支持：

- 会话查询
- 工作台查询

基础操作应包括：

- 列出运行中的流程
- 查看某个流程当前状态
- 暂停某个流程
- 继续某个流程
- 取消某个流程

这些能力在技术上可以通过 tool 或 skill 暴露给 Agent，但在产品上它们属于引擎能力，而不是会话临时技巧。

## 11. 设计原则

### 11.1 模板优先

用户最终需要的是“做一类事情的方法”，不是单个节点。

### 11.2 长时运行优先

所有模型与交互都应假设任务可能持续很久，而不是一次会话完成。

### 11.3 工件优先

重要成果必须沉淀为工件，而不是只停留在聊天记录里。

### 11.4 双入口优先

会话与工作台都必须是一等入口。

### 11.5 治理优先

引擎的价值不在于“能跑”，而在于“能被控制、被审计、被恢复、被纠偏”。

## 12. 开发要求

以下要求用于约束后续开发方向。

### 12.1 命名要求

对外主名称统一为：

`Agent 编排引擎`

`workflow` 仅用于表示工作台或底层模板表达层，不应再作为整个产品的总称。

### 12.2 入口要求

必须存在两类入口：

- 会话入口
- 工作台入口

且二者可操作同一对象。

### 12.3 模板要求

模板至少需要支持：

- 创建
- 编辑
- 保存
- 运行
- 从已有模板派生

### 12.4 运行时要求

运行时至少需要支持：

- 启动
- 暂停
- 继续
- 取消
- 查询运行中列表
- 查询单个实例状态

### 12.5 会话集成要求

会话至少需要支持下列意图被 Agent 正确调用：

- 启动一个模板
- 根据会话上下文匹配模板
- 创建一个模板草稿
- 列出运行中的流程
- 暂停一个流程
- 继续一个流程

## 13. 验收标准

以下条目满足后，才可认为一期达到本文档要求。

### 13.1 名称与定位验收

- 产品文档中对外名称统一为 `Agent 编排引擎`
- `workflow` 页面定位明确为“模板与运行管理工作台”

### 13.2 会话入口验收

满足以下场景：

1. 用户在会话中描述目标
2. 用户要求启动执行
3. Agent 能匹配或指定模板
4. 系统成功创建流程实例
5. 用户可在会话中查询该实例状态

### 13.3 模板生成验收

满足以下场景：

1. 用户在会话中要求参考已有模板生成新模板
2. Agent 生成新的模板草稿
3. 新模板可在工作台中被查看和编辑

### 13.4 工作台验收

工作台必须能完成：

- 查看模板列表
- 点击进入模板编辑
- 创建模板
- 手动运行模板
- 查看运行中的流程
- 暂停流程
- 继续流程
- 取消流程

### 13.5 状态一致性验收

在会话中暂停的流程，在工作台必须立即可见为暂停状态。

在工作台中继续的流程，在会话中查询时必须返回最新状态。

### 13.6 引擎能力验收

至少存在一套可运行模板，满足以下条件：

- 可被会话启动
- 可被工作台启动
- 可进入运行中状态
- 可被暂停
- 可被继续
- 可被取消

## 14. 示例模板

一期至少建议准备两类示例模板，用于校验引擎是否真的具备通用性。

### 14.1 小说模板

建议至少有：

- 起点小说模板
- 番茄小说模板

两者之间应能体现模板差异，而不是只有名字不同。

### 14.2 软件模板

建议至少有：

- 软件需求到开发的基础模板

它用于验证引擎并非只服务于内容创作场景。

## 15. 当前工作的正确演进方向

现有 `workflow lite` 已经完成的能力，例如：

- `run / pause / resume / retry`
- `llm / if / loop`
- checkpoint
- execution stack
- run view

仍然是有价值的。

它们应继续保留，但角色需要上升：

- 作为 `Agent 编排引擎` 的底层执行能力
- 作为模板工作台的实现基础
- 作为阶段内部流程的 DSL 与 runtime

因此，下一步不是推翻 workflow，而是把它纳入 `Agent 编排引擎` 的总体设计中。

## 16. 结论

Rhythm 接下来要建设的，不是一个更像 `n8n` 的工具平台，而是一个：

- 支持会话驱动
- 支持工作台驱动
- 支持模板创建与模板生成
- 支持长期运行与状态管理
- 支持暂停、继续、取消与查询
- 支持多领域任务方法论复用

的 `Agent 编排引擎`。

其中：

- 会话负责理解、生成、调用与控制
- 工作台负责模板编辑与运行管理
- 引擎负责统一执行、持久化与治理

这就是后续开发和验收应对齐的目标形态。
