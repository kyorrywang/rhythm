# Rhythm 前端会话线功能清单

本文档按页面区域整理桌面主界面围绕“会话”主线应具备的前端功能，包含展示、交互、状态和未来扩展方向。

说明：
- 这是一份产品功能文档，不是实现对照表。
- 文中“应”表示目标行为或期望交互。
- 本文档已经将 `Workbench`、插件入口、设置入口纳入整体布局。

## 1. 左侧全局栏

位置说明：
- 位于界面最左侧，窄栏。
- 承载全局级导航，而不是单个会话的局部操作。

### 1.1 工作区入口列表

功能定义：
- 顶部的 `R`、`N` 只是示例。
- 每一个按钮代表一个已打开的文件夹工作区。
- 按钮文案应使用文件夹名称的首字母缩写。
- 当前激活的文件夹应有高亮态。

交互规则：
- 点击某个文件夹按钮：
  - 如果点击的是非当前激活文件夹，应切换到该文件夹，并展开其右侧对应的会话侧边栏。
  - 如果点击的是当前激活文件夹，应切换展开/收起状态。
- 收起时：
  - 左侧全局栏继续保留。
  - 原会话侧边栏收起。
- 再次点击当前工作区：
  - 若当前为收起状态，则重新展开。
- hover 时应显示完整文件夹名称或完整路径提示。

状态要求：
- 支持激活态。
- 支持收起态。
- 支持 hover 态。
- 支持状态点或提醒点，但其语义必须明确。

### 1.2 添加工作区按钮

功能定义：
- `+` 按钮用于添加新的文件夹工作区，不是新建会话。

交互规则：
- 点击后触发“打开文件夹/添加文件夹”的流程。
- 添加成功后：
  - 新工作区出现在左侧全局栏。
  - 可选择自动切换到该工作区。
- 用户取消选择时，不新增任何项。

### 1.3 插件入口

功能定义：
- 设置入口上方应增加一个“插件”入口。
- 插件入口属于全局导航。

交互规则：
- 点击插件入口后，左侧原本的会话列表区域切换为插件列表区域。
- 插件入口应有激活态。
- 点击某个插件后，Workbench 出现，用于展示插件详情。
- Workbench 可以与 Chat 并排，也可以覆盖 Chat，具体由布局模式控制。

### 1.4 设置入口

功能定义：
- 左下角齿轮按钮用于打开设置模式。

交互规则：
- 点击设置后，左侧原本的会话列表区域切换为设置导航列表。
- 点击某个设置项后，Workbench 出现，用于展示具体设置页面。
- 设置入口应有激活态。
- 支持快捷键打开，例如 `Ctrl+,`。

## 2. 左侧内容区

位置说明：
- 位于全局栏右侧。
- 在不同模式下复用同一个区域。

区域模式：
- 会话列表模式
- 插件列表模式
- 设置列表模式

切换规则：
- 默认显示会话列表模式。
- 点击全局栏中的插件入口，切换为插件列表模式。
- 点击全局栏中的设置入口，切换为设置列表模式。
- 从插件或设置退出后，回到会话列表模式。

### 2.1 会话列表模式

#### 2.1.1 顶部项目头部

功能定义：
- 顶部展示当前文件夹名称。
- 顶部展示当前文件夹路径。
- 顶部右侧应有一个三个点的更多按钮。

展示要求：
- 文件夹名称为主标题。
- 路径为次级信息，超长时截断。
- hover 时显示完整路径。

交互规则：
- 点击三个点按钮后打开工作区级菜单。

工作区级菜单建议包含：
- 在系统文件管理器中打开
- 复制路径
- 刷新会话列表
- 收起侧边栏
- 从工作区列表移除

#### 2.1.2 新建会话按钮

功能定义：
- 项目头部下方应有“新建会话”按钮。

交互规则：
- 点击后新建会话并切换到该会话。
- 若当前正在运行其他会话，是否允许直接切走由产品策略决定。

#### 2.1.3 会话列表

功能定义：
- 展示当前工作区下的所有会话。
- 列表按更新时间排序，默认最近更新在上。
- 会话标题应自动生成，不应长期使用 session id。

每个会话项应包含：
- 左侧状态区域
- 中间标题区域
- 右侧时间/操作区域

#### 2.1.4 左侧状态区域

功能定义：
- 会话运行中时，左侧显示转圈圈的 loading 图标。
- 空闲状态时，左侧显示简洁状态标记。
- 当某个会话已经完成，但用户当前不在该会话内时，标题左侧显示一个小蓝点，表示有新的完成结果待查看。

交互规则：
- 鼠标悬停在会话项上时，左侧状态区域切换为钉子图标。
- 钉子图标用于 pin / unpin 会话。
- 点击钉子后，会话进入置顶状态。
- 小蓝点在用户点击进入该会话后消失。
- 仅 hover 不应消除小蓝点。

状态要求：
- 支持 idle、running、pinned、completed_unread、hover 等多种视觉态。

#### 2.1.5 中间标题区域

功能定义：
- 标题应由系统根据首条消息或会话主题自动生成。
- 标题应短、稳定、可扫描。
- 标题过长时截断显示。

交互规则：
- hover 时可展示完整标题。
- 后续可扩展支持手动重命名。

#### 2.1.6 右侧时间/操作区域

功能定义：
- 默认显示相对更新时间。

更新时间格式建议：
- 刚刚
- x分钟前
- x小时前
- x天前
- 日期

交互规则：
- 鼠标悬停在会话项上时，右侧时间位置切换为三个点的更多按钮，而不是直接显示归档按钮。
- 点击更多按钮后，打开该会话的快捷操作菜单。

会话项更多菜单建议包含：
- 归档会话
- 重命名会话
- 复制会话标题
- 复制会话 ID
- 删除会话

#### 2.1.7 会话项点击行为

功能定义：
- 点击会话项主体应切换到该会话。
- 当前激活会话应有高亮样式。

交互规则：
- 点击已激活会话时，可保持当前状态，不重复刷新。
- 若该会话存在未读完成蓝点，点击进入后蓝点消失。

#### 2.1.8 列表扩展能力

建议支持：
- pinned 分组
- archived 分组
- 搜索会话
- 按日期分组
- 按主会话 / 子会话分层
- 快速定位未读完成会话

### 2.2 插件列表模式

功能定义：
- 当用户点击全局栏中的插件入口时，该区域显示插件列表，而不是会话列表。

列表项应展示：
- 插件名称
- 版本号
- 简短描述
- 启用 / 禁用状态
- 技能数量

交互规则：
- 点击某个插件项后，Workbench 打开插件详情页。
- 当前选中的插件项应有高亮态。
- 建议支持搜索和启用状态过滤。

### 2.3 设置列表模式

功能定义：
- 当用户点击全局栏中的设置入口时，该区域显示设置导航列表，而不是会话列表。

建议设置分类：
- 模型
- 会话
- 权限
- 记忆
- Hooks
- MCP
- 自动压缩
- 插件
- 定时任务
- 前端显示

交互规则：
- 点击某个设置项后，Workbench 打开对应的设置页面。
- 当前选中的设置项应有高亮态。

## 3. Workbench 区域

位置说明：
- 位于左侧内容区与聊天主区域之间。
- 默认隐藏。

功能定位：
- Workbench 是会话工作台侧板。
- 消息流负责记录过程。
- Workbench 负责展示对象本身、配置详情或结果视图。

布局模式：
- `split`
  - Workbench 与 Chat 并排显示，适合一边看消息一边看结果。
- `focus`
  - Workbench 覆盖或替代 Chat 区域，适合插件页、设置页、网页预览等完整视图。

交互规则：
- 默认收起。
- 用户点击某个文件、某个插件、某个设置项、某个预览入口时，Workbench 打开。
- 建议支持 tab 化管理多个 Workbench item。
- 建议支持 pin、close、restore。
- 建议支持“跟随会话”和“固定全局”两种模式。

建议支持的内容类型：
- File
- Diff
- Web
- Generated
- Plugin
- Settings
- Task Output

### 3.1 文件类 Workbench

应支持：
- 打开文件内容
- 展示文件路径
- 只读预览
- 后续可扩展跳转行号、复制路径、在外部打开

### 3.2 Diff 类 Workbench

应支持：
- 展示 edit / write 的差异结果
- 展示新增 / 删除统计
- 展示 patch 或更清晰的对比视图

### 3.3 Web 类 Workbench

应支持：
- 展示网页或本地服务预览
- 展示加载状态
- 展示地址信息
- 刷新
- 在系统浏览器中打开

### 3.4 插件类 Workbench

应支持展示：
- 插件基本信息
- 启用状态
- 技能数量
- 安装路径
- 插件 hooks
- 插件 MCP 能力
- 安装 / 卸载 / 启用 / 禁用操作

### 3.5 设置类 Workbench

应支持展示：
- 左侧导航选中的具体设置页面
- 表单编辑
- 保存、取消、恢复默认等操作

## 4. 会话主区域头部

位置说明：
- 位于主会话区域最上方。
- 随滚动吸顶。

### 4.1 会话标题显示

功能定义：
- 显示当前会话标题。
- 若为子会话，应体现其所属关系。

### 4.2 返回主会话

功能定义：
- 当前会话存在 `parentId` 时，应显示返回主会话入口。

交互规则：
- 点击后返回父会话。
- 后续可扩展为 breadcrumb。

### 4.3 上下文用量入口

功能定义：
- 右上角应有上下文用量按钮，建议使用圆环进度图形表示。

交互规则：
- 点击打开上下文用量面板。
- hover 时显示 tooltip。
- 圆环进度应与真实 token / context 数据联动。

### 4.4 更多操作按钮

功能定义：
- 右上角应有三个点或更多操作按钮。

建议菜单项：
- 重命名会话
- 复制会话 ID
- 导出会话
- 清空消息
- 归档会话
- 删除会话

### 4.5 错误和限制状态

功能定义：
- 会话出错时显示错误横幅。
- 达到最大轮次时显示警告组件。

交互规则：
- 错误横幅支持 dismiss。
- 最大轮次警告可扩展继续操作入口。

---

## 5. 消息流区域

位置说明：
- 主会话页面中部。
- 承载整个会话时间线。

### 5.1 用户消息

功能定义：
- 展示用户发送的消息。
- 支持普通输入、追加输入、Ask 回答形成的消息回显。

建议扩展：
- 复制用户消息
- 重新编辑并发送
- 基于该条消息新开分支会话

### 5.2 AI 消息

功能定义：
- AI 消息由多个 segment 构成。

主要 segment 类型：
- thinking
- tool
- ask
- text
- permission

### 5.3 Thinking 段

功能定义：
- 展示模型思考过程。
- 支持流式更新。
- 支持折叠 / 展开。
- 支持耗时显示。

交互规则：
- 点击段标题切换展开 / 收起。
- 默认展开状态受显示设置控制。

### 5.4 Tool 段

功能定义：
- 不同 tool 应使用不同的文案模板和信息结构。
- 标题行应高可扫描，尽量在不展开的情况下就能看懂动作和对象。

统一要求：
- 保持稳定命名风格。
- 信息顺序尽量固定。
- 超长命令和路径截断显示，hover 可查看完整内容。
- 对于有明确结果对象的 tool，应提供“在 Workbench 中打开”的入口。

#### 5.4.1 Shell

展示建议：
- 标题格式接近：`Shell <命令>`
- 尾部显示耗时，例如 `Shell npm test (6s)`

展开后应展示：
- stdout
- stderr
- 退出状态
- 完整命令

与 Workbench 联动：
- 如果启动了本地服务或返回可预览地址，应支持一键在 Workbench 中打开网页。

#### 5.4.2 Read

展示建议：
- 标题格式接近：`Read <path>`

展开后应展示：
- 读取摘要
- 文件内容预览

与 Workbench 联动：
- 点击 path 或操作按钮后在 Workbench 中打开文件。

#### 5.4.3 Write

展示建议：
- 标题格式接近：`Write <path>`
- 如能拿到写入统计，可补充规模信息。

展开后应展示：
- 写入摘要
- 具体写入内容或写入结果

与 Workbench 联动：
- 写入完成后可在 Workbench 中打开最终文件。

#### 5.4.4 Edit

展示建议：
- 标题格式接近：`Edit <path>`
- 附带 diff 摘要，例如 `+1 -0`、`+12 -3`

展开后应展示：
- patch
- 变更详情

与 Workbench 联动：
- 默认优先在 Workbench 中打开 diff 或编辑结果。

#### 5.4.5 Delete

展示建议：
- 标题格式接近：`Delete <path>`

展开后应展示：
- 删除结果
- 影响范围
- 日志

与 Workbench 联动：
- 可在 Workbench 中展示删除结果摘要，而不是文件内容。

#### 5.4.6 Spawn Subagent

展示建议：
- 标题体现子代理或子会话行为。
- 摘要显示子任务主题。

交互规则：
- 若已存在子会话 ID，应支持点击进入子会话。

### 5.5 Ask 段

功能定义：
- 展示模型发出的追问。
- 支持单问题或多问题。
- 支持显示回答结果。

交互规则：
- 点击段标题可展开 / 收起。
- 回答动作在底部 Ask Dock 进行。
- 回答完成后，时间线中应显示：
  - 选中的选项
  - 补充说明
  - 回答耗时
  - 被忽略的问题状态

### 5.6 Text 段

功能定义：
- 展示最终文本输出。
- 支持 Markdown。
- 支持代码块高亮。
- 支持 GFM 能力。

建议扩展：
- 复制整条回答
- 针对代码块单独复制
- 折叠超长内容

### 5.7 权限事件段

功能定义：
- 权限申请不应只出现在全局弹窗里，也应沉淀到时间线中。

应展示：
- 触发的工具
- 申请原因
- 当前状态：等待中 / 已允许 / 已拒绝

### 5.8 特殊系统事件

建议纳入时间线：
- 上下文压缩
- 定时任务触发
- 定时任务完成
- 中断
- 最大轮次到达

### 5.9 消息级 footer 操作

建议功能：
- 复制消息
- 显示模型来源
- 显示总耗时
- 重试
- 继续生成

## 6. 空态区域

位置说明：
- 没有 active session 或 active session 还没有消息时展示。

功能定义：
- 提供欢迎文案和产品定位说明。
- 底部 Composer 在空态下仍正常显示，可以直接发送消息。

交互规则：
- 不应要求用户必须先手动点击“新建会话”才能发送第一条消息。
- 用户在空态发送第一条消息时，应自动创建一个新会话。
- 自动建会话流程应包括：
  - 创建会话
  - 将新会话加入左侧会话列表
  - 自动切换到该会话
  - 立即开始流式对话
- 若自动建会话失败，应给出明确错误反馈，并保留输入内容。

建议扩展：
- 推荐 prompt
- 最近会话快捷入口
- 当前工作区说明

---

## 7. 底部 Composer 区

位置说明：
- 固定在主会话区域底部。
- 根据会话 phase 切换不同 dock 形态。

### 7.1 主输入模式

功能定义：
- 默认展示文本输入框。
- 支持多行输入。
- 支持发起新一轮消息。
- 输入区应拆成“框内内容区”和“框外控制条”两层。

交互规则：
- 空文本不应发送。
- 发送后清空输入框。
- 空态和普通会话态都可用。

#### 7.1.1 框内内容区

功能定义：
- 输入框左下角应有一个 `+` 按钮。
- 用户可以通过这个入口插入文件或图片。
- 用户也可以直接通过粘贴的方式把文件或图片加入输入区。
- 插入后的内容应显示在输入框内部，而不是脱离输入区单独漂浮。

支持的输入对象：
- 本地文件
- 图片文件
- 剪贴板图片
- 剪贴板文件

展示要求：
- 插入对象应以 attachment chip、卡片或缩略预览的形式显示在输入框内。
- 图片应显示缩略图。
- 文件应显示文件名、类型、大小或路径摘要。
- 多个附件应支持并排或换行展示。

交互规则：
- 点击 `+` 按钮后，打开插入菜单或系统文件选择器。
- 插入成功后，附件出现在当前输入框内。
- 用户应能删除某个已插入附件。
- 粘贴图片或文件时：
  - 若当前输入框可接收焦点，则应直接插入到当前消息草稿中。
  - 不应被误当成纯文本粘贴。
- 附件应与文本同属一条待发送消息。

建议扩展：
- 支持拖拽文件到输入框
- 支持插入后的重排
- 支持查看大图或文件详情

#### 7.1.2 框外控制条

位置说明：
- 位于输入框外部、下方。
- 属于当前输入消息的发送控制区。

从左到右建议包含四个主控件：
- mode
- model
- thinking
- permission

#### 7.1.3 Mode 选择器

功能定义：
- 第一个控件是模式选择器。
- 默认至少支持三种模式：
  - 正常
  - 计划
  - 协调

命名建议：
- 正常：`Chat` 或 `Normal`
- 计划：`Plan`
- 协调：`Coordinate`、`Orchestrate` 或 `Swarm`

说明：
- “协调”模式用于多代理、协作式、编排式任务，名称应体现协作而不是单纯聊天。

交互规则：
- 点击 mode 控件后弹出模式列表。
- 切换 mode 后，后续发送的消息应带着该模式语义进入后端流程。
- 当前 mode 应有明显选中态。

自定义模式：
- 支持用户自定义 mode。
- 自定义 mode 应区分两级作用域：
  - 项目级
  - 全局级
- 项目级 mode 仅在当前工作区可见。
- 全局级 mode 在所有工作区可见。

自定义 mode 建议支持：
- 名称
- 描述
- 图标或颜色标识
- 适用范围（项目 / 全局）

#### 7.1.4 Model 选择器

功能定义：
- 第二个控件是模型选择器。
- 点击后可选择当前消息使用的模型。

交互规则：
- 当前选中的模型应在控件上可见。
- 打开后可按 provider 分组显示模型列表。
- 切换模型只影响当前会话或当前消息，具体作用范围应明确。

建议展示信息：
- provider 名称
- model 名称
- 可选的标签，例如默认 / 快速 / 强推理

#### 7.1.5 Thinking 强度选择器

功能定义：
- 第三个控件是思考强度选择器。
- 至少支持：
  - low
  - medium
  - high

交互规则：
- 点击后切换思考强度。
- 当前强度应有选中态。
- 强度选择应影响后端推理深度或相关执行策略。

#### 7.1.6 Permission 开关

功能定义：
- 最后一个控件是权限开关。
- 开启后表示当前会话或当前执行上下文进入“全部允许”模式，不再逐次询问权限。

交互规则：
- 开启时，应明确提示这是高权限模式。
- 开启后，后续工具执行默认直接通过，不再弹权限确认。
- 关闭后恢复正常权限询问流程。

说明：
- 该开关应与后端的 permission mode 联动，而不是仅做前端视觉状态。
- 如果产品后续支持更细粒度权限策略，这里仍可保留为快捷入口。

### 7.2 Ask Dock

功能定义：
- 当会话进入 `waiting_for_ask` 时，Composer 切换为 Ask Dock。

Ask Dock 仅支持两类回答模式：
- 单选 + 补充输入
- 多选 + 补充输入

多问题流程：
- 多个问题不应一次性堆叠在一个面板中。
- 应采用分步式交互。
- 用户点击“下一步”后进入下一题。
- 到最后一题时，主按钮切换为提交。
- 每一题都允许填写补充说明。
- 用户点击忽略时，应跳过当前题并继续往下走，而不是终止整个 Ask 流程。
- 所有问题处理完成后，再统一提交给后端并回写时间线。

建议补充：
- 显示当前进度，例如 `第 1 / 3 题`
- 支持返回上一题修改答案
- 对不合法输入给出校验提示

### 7.3 Task Dock

功能定义：
- 展示当前任务列表。
- 每个任务有状态：
  - pending
  - running
  - completed
  - error

交互规则：
- 支持最小化 / 展开。
- 任务全部完成后可自动最小化。
- 最小化时应显示为一行摘要，而不是完全隐藏。

最小化摘要至少应包含：
- 当前任务标题或阶段标题
- 当前进度，例如 `2/5`

展开态要求：
- 支持最大高度限制。
- 任务数量过多时，列表区域内部应出现滚动条，而不是无限顶高 Composer。
- 应显示任务状态图标。
- 应清楚标识当前运行中的任务。
- 应清楚标识错误任务。

### 7.4 Append Dock

功能定义：
- 当会话正在运行，用户继续输入时，新输入不立即打断当前生成，而是进入队列。

交互规则：
- 显示队列长度。
- 展示每条排队消息摘要。
- 支持删除单条排队消息。
- 支持清空全部排队消息。
- 支持 interrupt。
- 支持最小化 / 展开。

建议扩展：
- 编辑排队中的消息
- 调整队列顺序
- 区分 urgent / normal 优先级

## 8. 会话状态机

前端应感知的核心 phase：
- idle
- streaming
- streaming_with_queue
- processing_queue
- waiting_for_ask
- interrupting
- waiting_for_permission

每个 phase 对 UI 的影响应明确：
- `idle`
  - 正常输入
  - 不显示运行中状态
- `streaming`
  - 消息流继续更新
  - 运行中计时显示
  - 新消息进入队列
- `streaming_with_queue`
  - 展示 Append Dock
- `processing_queue`
  - 当前轮完成后消费队列
- `waiting_for_ask`
  - 切换为 Ask Dock
- `interrupting`
  - 展示中断中状态
- `waiting_for_permission`
  - 打开权限弹层
  - 时间线同步记录权限事件

交互规则：

建议扩展：
- context limit
- 压缩前后变化
- 哪些部分最占上下文

## 10. 权限确认弹层

位置说明：
- 收到权限请求时，在主界面上方弹出。

功能定义：
- 展示权限申请的工具名和申请原因。
- 支持允许和拒绝。
- 支持“始终允许此工具”的策略设置。

交互规则：
- 点击允许：
  - 调用后端批准
  - 更新会话状态
  - 关闭弹层
- 点击拒绝：
  - 调用后端拒绝
  - 更新会话状态
  - 关闭弹层
- 勾选“始终允许此工具”后，应真正写入权限配置，而不是只停留在 UI 状态。

## 11. 子会话与子代理

功能定义：
- 支持从主会话中派生子会话。
- 子会话拥有自己的消息流和状态。
- 子会话记录父会话关系。

交互规则：
- 在工具段中可点击进入子会话。
- 在子会话头部可返回主会话。
- 必要时支持队列在父子会话之间迁移。

建议扩展：
- 子会话树结构展示
- 父子 breadcrumb
- 子会话结果摘要回灌父会话

## 12. 插件页

位置说明：
- 由全局栏中的插件入口进入。
- 左侧内容区显示插件列表。
- 点击某个插件后，Workbench 显示插件详情。

插件列表应展示：
- 插件名称
- 版本
- 描述
- 启用状态
- 技能数量
- 安装路径

插件详情页建议包含以下模块。

### 12.1 基本信息

应展示：
- 插件名称
- 版本
- 描述
- 安装路径
- 是否启用
- 是否默认启用

### 12.2 启用状态管理

应支持：
- 启用插件
- 禁用插件

后端依据：
- 已存在启用 / 禁用插件能力。
- 启用状态会写入后端 settings。

### 12.3 技能信息

应展示：
- skills 数量
- 后续可扩展为技能列表和技能说明

### 12.4 Hooks 信息

应支持展示插件贡献的 hooks，至少区分：
- pre_tool_use
- post_tool_use
- session_start
- session_end

每个 hook 建议展示：
- 类型
  - command
  - http
- timeout
- matcher
- 是否 block on failure

### 12.5 MCP 能力

应支持展示：
- 插件带来的 MCP server 配置
- server 名称
- 基本连接信息

### 12.6 插件安装与卸载

应支持：
- 从本地目录安装插件
- 卸载插件

交互规则：
- 安装成功后，插件列表应刷新。
- 卸载前需要确认。
- 卸载后若当前 Workbench 正在显示该插件，应自动回退。

### 12.7 搜索与过滤

建议支持：
- 按名称搜索
- 按启用 / 禁用过滤

## 13. 设置页

位置说明：
- 由全局栏中的设置入口进入。
- 左侧内容区显示设置导航列表。
- 点击某个设置项后，Workbench 显示具体设置页面。

设置分类应至少覆盖后端已有能力：
- 模型
- 会话
- 权限
- 记忆
- Hooks
- MCP
- 自动压缩
- 插件
- 定时任务
- 前端显示

### 13.1 模型设置

功能定位：
- 模型设置不应只保存一个 provider + 一个 model。
- 应支持多个 provider 配置，以及每个 provider 下维护多个可选模型。

provider 管理建议包含：
- provider name
- base_url
- api_key

交互规则：
- 支持新增 provider
- 支持编辑 provider
- 支持删除 provider
- 支持设置默认 provider

provider 示例：
- OpenAI
- Anthropic
- 其他兼容 OpenAI API 的服务

每个 provider 下的 model 管理建议包含：
- model name
- 是否默认
- 是否可用
- 可选的备注信息

应支持的模型管理操作：
- 新增 model
- 编辑 model
- 删除 model
- 设置默认 model

示例说明：
- OpenAI provider 下可以配置多个 model，例如：
  - `chatgpt-3.5`
  - `gpt-5.4`
  - 其他用户自定义名称
- Anthropic provider 下同样可以维护多模型列表。

交互规则：
- 修改后支持保存到后端 settings。
- `api_key` 应做安全显示处理。
- 模型选择器应与 Composer 下方的 model 控件联动。
- 当前默认 provider / 默认 model 应能被会话界面直接使用。

建议补充：
- provider 连通性校验
- model 可见性开关
- 测试请求按钮

### 13.2 会话设置

建议包含：
- max_turns
- system_prompt

### 13.3 权限设置

建议包含：
- permission mode

交互规则：

- 支持配置禁止命令。

### 13.4 记忆设置

### 13.5 Hooks 设置

建议包含：

每类 hook 建议支持：
- 新增

### 13.6 MCP 设置

后续可扩展：

### 13.8 插件设置
